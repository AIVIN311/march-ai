# March / Match AI
## Project Brief

### 一句話版本

**March / Match AI** 不是滑卡交友軟體，而是一個 **先理解、再介紹** 的關係提案系統；它用授權、記憶、摘要與低頻提案，讓相遇從一段真實開始，而不是從外貌與無限選擇開始。

---

## 1. 這個專案在做什麼

這個專案要驗證的，不是「AI 能不能幫人配對」這種很泛的命題，而是更精準的兩件事：

1. **以「被理解」開始，是否比傳統交友流程更有感。**
2. **被系統理解的感覺，能不能做到溫柔而不侵入。** 

換句話說，這不是在做一個更聰明的 Tinder，  
而是在設計一種 **更不浪費人的相遇方式**。

---

## 2. 核心產品觀

這個產品的底層不是市場邏輯，而是**理解邏輯**。  
它的主要原則已經相當清楚：

* 先理解，再介紹
* 授權先於理解
* 原始輸入不直接成為媒合層
* 提案不是判決，而是一段可接受、可稍後、可拒絕的介紹
* 外貌不是入口，真實與節奏才是入口
* 使用者有權修正、撤回、校準被理解的方式

---

## 3. 產品不是只有一種模式

March / Match AI 並不是只做「戀愛媒合」。  
你現在其實已經把它看成一顆 **共同引擎**，上面可以長出三種模式：

* **相遇模式**：讓兩個人被有理由地介紹
* **共研模式**：讓兩個人因為某個問題、想法、研究興趣而被配對
* **共創模式**：讓能合作、能一起扛壓力、能共做事情的人被配對

所以戀愛不是唯一應用，它只是最先驗證的入口場景。

---

## 4. 第一版的實作哲學

第一版不是做完整 AI agent 生態，而是做 **夠真、夠輕、夠可測** 的 MVP。  
目前方向已經很明確：

* 先不做每個人一個 agent
* 採用共享模型服務 + 每個人自己的記憶層 / 授權層
* 先保留前端 heuristic / preset 來模擬 intelligence
* 先不接後端、登入、照片交換、外部聯絡方式

這讓專案保持可落地，不會一下子變成重系統。

---

## 5. 市場與語言策略

你已經做出一個很關鍵的決定：**Day 1 英文，Global First**。  
原因不是只有語言翻譯，而是：

* 英文市場更適合這種新型相遇協議的初次測試
* 冷啟動市場基數大得多
* 你在產品命名、產品哲學、未來敘事上，本來就已經習慣用英文思考
* AI 在英文世界的處理穩定度與精度也較有利於第一版原型驗證 

所以目前前台方向是 **English-first UI**，而不是雙語前台。

---

## 6. 品牌與語氣

品牌主體是 **March**，**Match AI** 是 descriptor。  
首頁不是先問問題，而是先建立一種安靜、可信、非市場式的氣味。  
目前已經鎖定的語氣鏈大致是：

* Landing hero：**Not everyone needs to meet everyone.**
* Onboarding opening：**You don’t need to impress anyone here.**

這條語氣鏈的功能不是漂亮，而是讓使用者從一開始就感覺：  
這裡不是在叫你表演自己，而是在幫你先被理解。

---

## 7. Onboarding v0.2：現在最重要的實作核心

目前最新的產品核心已經不是一般 onboarding，而是 **inline onboarding v0.2**。  
它長在單頁 dark-journal prototype 裡，不切成 wizard、不做 modal，而是像一段在頁面裡慢慢長出的流程。核心 state 是：

`entry → consent → freeTalk → assist → reflection → summaryReview → inlineEdit → gapFill → complete → Today`

這裡最關鍵的不是 state 名稱，而是三件事：

1. 使用者先被安放，而不是先被審問
2. AI 先反映它聽到什麼，再交給使用者校準
3. 真正拿去做介紹的，不是 raw input，而是經過確認後的較安全層

---

## 8. 目前已經形成的三層資料觀

這個專案最成熟的地方之一，就是你已經把資料層分清楚了。  
目前可以理解成三層：

* **your original words**：原始輸入
* **your reviewed understanding**：經過使用者確認的理解摘要
* **what can be used for introductions**：更輕、更安全、不直接暴露原始脆弱內容的介紹層 

這個分層是倫理感、體驗感與未來真實 AI 化的重要基礎。

---

## 9. 你真正要配的，不是表面心情

這個專案的媒合邏輯，也不是看一個人快不快樂、外向不外向、喜歡什麼興趣而已。  
你前面已經抓到一個很高級的判斷：

> **不要用情緒表層做媒合，要用情緒代謝方式做媒合。**

所以系統真正要觀察的是：

* 情緒怎麼表達
* 需要怎樣被接住
* 語言溫度
* 關係節奏
* 修復能力
* 是在發洩、求救、還是在建立黑色幽默式連結

這會讓你的提案不只是「你們好像都愛吐槽」，  
而是「你們的痛苦與陪伴方式可能形成健康互動」。

---

## 10. 現在的技術與工作型態

這個專案目前最需要的，不是龐大 AI 工程堆疊，而是：

* HTML / CSS / JS
* 狀態流設計
* 產品互動文案
* 驗收與視覺主次判斷
* Git 版本管理
* Browser / Playwright / screenshot 類驗收工具 

也就是說，現在最難的不是後端，而是**分寸**。

---

## 11. 目前專案階段

如果用最實際的話來說，你現在已經不在「只有想法」的階段，  
而是在：

> **從規格討論期，跨進可體驗原型期。**

目前的重點不再是擴充功能，而是：

* 讓 preview-only framing 完整成立
* 拉大 current / receded 的視覺主次差
* 收掉 complete 與 entry fragment 的解釋味
* 讓整體更像 March，而不是高質感 SaaS wizard 

---

## 12. 這個專案現在最不該做的事

為了保住專案節奏，下面這些現在都不應該是優先級：

* 不應先做真實雙向媒合
* 不應先接 LLM / backend 讓事情變重
* 不應先做 auth / photo / messaging / notifications
* 不應因為 UI 有點不穩就整個重做 layout
* 不應讓 onboarding 重新膨脹成多頁 wizard

---

## 13. 一句最終定位

如果未來你要把整個專案濃縮成一句對外或對自己都成立的話，我會推薦這個版本：

> **March / Match AI is a relationship proposal system that begins with understanding, not performance.**

或者中文更貼近你本心的版本是：

> **March 不是讓人互相挑選，而是讓人先被理解，然後才被介紹。**
